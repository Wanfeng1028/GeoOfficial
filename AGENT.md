# GeoWork 官网项目 Agent 协作规范

> 适用范围：**仅适用于 GeoWork 官网项目**。  
> 不适用于 GeoWork 桌面软件、GeoFrontend2.0 桌面前端重构项目或其他仓库。

## 1. 项目目标

GeoWork 官网的核心目标是：

- 清晰说明 GeoWork 是什么、解决什么问题、适合哪些用户。
- 展示 GeoWork 在 GIS、遥感、空间数据与 AI Agent 方向的产品能力。
- 建立专业、可信、现代且统一的品牌形象。
- 保证所有页面、路由、交互和响应式布局真实可用。
- 保证亮色模式、暗色模式、构建、CI 和 E2E 稳定。

所有改动都必须服务于官网，不得混入桌面端工作台、Electron 窗口、终端面板、桌面侧边栏等无关内容。

## 2. 设计参考

中文字体、字号、行高、标题层级、段落密度和页面阅读节奏参考：

- FlowUs 产品官网：https://flowus.cn/product

重点参考：

- 中文标题的大小与字重。
- 正文可读性。
- 标题与正文之间的比例。
- 区块间距。
- 页面内容密度。
- 中文按钮和导航的视觉尺寸。

### 2.2 英文内容

英文标题、数字、留白、卡片边界和整体精致感参考：

- Attio 官网：https://attio.com/

重点参考：

- 英文标题字体比例。
- 字母间距。
- 英文与数字的视觉表现。
- 大区块留白。
- 卡片边框和背景层级。
- 克制、现代、高级的视觉风格。

### 2.3 参考原则

- 可以参考设计规律，但不得直接复制对方源码、文案、品牌资产或专有图形。
- 最终结果必须形成 GeoWork 官网自己的设计语言。
- 不得只参考首页，必须查看参考网站的相关页面和不同内容类型。

## 3. 强制执行规则

### 3.1 只处理官网

- 所有任务、方案、文档和代码都必须围绕官网项目。
- 不得加入桌面软件的窗口控制、终端、日志面板、Electron、桌面侧边栏等设计内容。
- 不得把 GeoFrontend2.0 的页面结构直接套到官网。
- 如果发现需求属于其他项目，必须先明确指出，不得直接混改。

### 3.2 每次只完成一个小阶段

- 每次只处理一个明确、可验证的小问题。
- 不要一次修改大量无关文件。
- 不要在没有必要时大范围重构。
- 完成一小部分后，先运行、检查、截图，再进行下一部分。
- 除非明确要求一次性输出全部内容，否则按步骤推进。

### 3.3 必须检查全部路由

任何涉及以下内容的改动：

- 全局字体。
- 全局颜色。
- 主题。
- 导航栏。
- 页脚。
- 页面容器。
- 公共按钮。
- 公共卡片。
- 响应式规则。
- 全局 CSS。
- 设计 Token。

都不能只检查首页。

必须：

1. 找到官网完整路由表。
2. 列出全部公开页面。
3. 逐个打开并检查。
4. 检查正常页面、404 页面、空状态、加载状态和错误状态。
5. 搜索公共组件的全部引用位置。
6. 确认修改没有破坏其他页面。

> 不能只停留在首页，必须检查官网的所有页面和所有路由。

### 3.4 每完成一小部分都要浏览器截图

每完成一个小阶段，必须：

1. 启动官网项目。
2. 在真实浏览器中打开修改页面。
3. 检查修改前后的区别。
4. 截图保存结果。
5. 检查桌面宽屏、普通桌面和窄屏。
6. 检查亮色模式与暗色模式。
7. 检查浏览器控制台。
8. 确认没有黑屏、白边、溢出、错位、闪烁和布局跳动。

不得只根据代码推断“应该已经修好”。

> 每完成一小部分，都必须去浏览器里面截图，检查实际效果是否符合预期。

### 3.5 每次改动后必须通过 CI 和 E2E

每次代码改动完成后，必须确保：

- 依赖安装成功。
- Lint 通过。
- TypeScript 或对应类型检查通过。
- 单元测试通过。
- 构建通过。
- CI 通过。
- E2E 通过。
- 浏览器控制台没有新增错误。
- 页面资源和网络请求没有新增失败。
- 所有受影响路由可以正常访问。

如果仓库暂时没有某项检查，必须明确说明，不能伪造结果。

> 每次改动完所有代码，都必须确保 CI 和 E2E 通过。

## 4. 修改前必须检查的内容

开始修改前，至少检查：

- `package.json` 中的脚本。
- 项目入口。
- 路由配置。
- 全局布局组件。
- 顶部导航。
- 页脚。
- 全局样式文件。
- 主题实现。
- 字体加载方式。
- 设计变量或 Token。
- 当前使用的 UI 组件库。
- CI 工作流。
- E2E 配置。
- 受影响页面和组件。
- 当前浏览器实际效果。

禁止只看一张截图后直接覆盖样式。

---

## 5. 代码修改原则

### 5.1 保持现有技术路线

- 优先使用项目当前已有的框架、组件库和样式方案。
- 未经明确同意，不引入新的大型 UI 框架。
- 未经明确同意，不迁移到 Tailwind CSS。
- 不重复创建已有功能的组件。
- 不为简单样式修改增加不必要依赖。
- 新增依赖前必须说明用途、体积、维护情况和替代方案。

### 5.2 优先修复根因

- 不用大面积遮罩掩盖黑屏问题。
- 不用固定高度制造空白。
- 不用随意的负 margin 修复结构。
- 不用大量 `!important` 强行覆盖。
- 不用关闭类型检查、Lint 或测试掩盖错误。
- 不通过删除测试让 CI 通过。
- 不用临时假数据长期代替真实页面逻辑。

### 5.3 改动范围必须清晰

每次汇报必须写明：

- 文件完整路径。
- 修改位置或搜索关键词。
- 原来的问题。
- 修改内容。
- 修改原因。
- 影响的页面和路由。
- 浏览器验证结果。
- CI 与 E2E 结果。

如果需要用户手动修改，必须明确写出：

- 修改哪个文件。
- 定位到哪一段。
- 把哪段代码改成什么代码。

不能只给模糊建议。

## 6. 官网专项规则

### 6.1 顶部导航

- 导航高度、文字大小、Logo 大小和按钮尺寸必须协调。
- 所有导航项必须有真实路由或真实行为。
- 当前路由状态必须清晰。
- 移动端菜单必须可用。
- 导航滚动、吸顶、隐藏或显示行为必须稳定。
- 不能因主题切换出现闪烁或颜色残留。

### 6.2 首页

首页必须清楚表达：

- GeoWork 是什么。
- GeoWork 面向哪些人。
- GeoWork 的核心价值。
- GeoWork 的主要能力。
- 用户下一步可以做什么。

首页不能：

- 整页纯黑。
- 只有空洞口号。
- 存在巨大无意义空白。
- 按钮没有实际行为。
- 产品截图无法看清。
- 因固定高度造成内容断层。
- 页脚悬浮在页面中间。

### 6.3 内容页面

所有内容页必须：

- 使用统一的页面容器。
- 使用统一的标题层级。
- 使用统一的正文宽度。
- 使用统一的卡片、按钮和标签。
- 保证中英文排版一致。
- 保证亮色和暗色均可读。
- 保证窄屏下没有横向溢出。

### 6.4 页脚

- 页脚必须自然出现在内容底部。
- 内容较少时仍应位于视口底部。
- 不得使用错误的固定定位制造白条。
- 不得遮挡正文或按钮。
- 所有路由中的页脚风格和行为保持一致。

### 6.5 主题

- 官网所有路由必须统一支持亮色和暗色。
- 禁止出现同一页面部分亮、部分暗。
- 暗色不能只是使用纯黑背景。
- 所有文本、边框、卡片和按钮都要有主题状态。
- 主题变量必须集中维护。
- 刷新页面时不应明显闪烁。

### 6.6 响应式

至少检查：

- 1440px 及以上宽屏。
- 1280px 普通桌面。
- 1024px 窄桌面。
- 768px 平板宽度。
- 390px 左右移动端。

重点检查：

- 导航是否溢出。
- 标题是否异常断行。
- 卡片是否过窄。
- 按钮是否换行错位。
- 图片是否超出容器。
- 页脚是否异常。
- 是否出现横向滚动条。

## 7. 标准执行流程

### 第一步：确认任务

- 明确用户要修复的问题。
- 查看截图、报错和相关代码。
- 确认当前分支和提交。
- 确认只修改官网项目。
- 列出受影响页面与路由。

### 第二步：建立修改前基线

- 安装依赖。
- 启动项目。
- 打开受影响页面。
- 检查全部相关路由。
- 保存修改前截图。
- 记录控制台错误。
- 运行当前构建、测试和 E2E。

### 第三步：小步修改

- 只处理当前小阶段。
- 优先修复根因。
- 保持改动集中。
- 同步补充必要的类型与测试。
- 不进行无关重构。

### 第四步：浏览器验证

- 打开修改页面。
- 检查全部受影响路由。
- 检查亮色模式。
- 检查暗色模式。
- 检查不同窗口宽度。
- 检查控制台。
- 截图保存修改后效果。

### 第五步：工程验证

实际命令以项目 `package.json` 和 CI 配置为准，通常包括：

```bash
npm ci
npm run lint
npm run typecheck
npm run test
npm run build
npm run e2e
```

不得机械执行不存在的命令，也不得伪造结果。

### 第六步：结果汇报

每次完成后必须汇报：

- 已完成事项。
- 修改文件。
- 修改的页面与路由。
- 浏览器截图验证结果。
- 亮色与暗色验证结果。
- 响应式验证结果。
- 构建、CI 和 E2E 结果。
- 当前仍存在的问题。
- 下一小步计划。

---

## 8. 禁止事项

- 禁止把桌面软件规则写进官网项目。
- 禁止只检查首页。
- 禁止伪造浏览器截图。
- 禁止伪造 CI、构建或 E2E 结果。
- 禁止删除不理解的代码。
- 禁止覆盖用户未提交的重要改动。
- 禁止提交密钥、Token、密码和私有接口。
- 禁止通过关闭规则掩盖问题。
- 禁止为追求视觉效果破坏可访问性和性能。
- 禁止直接复制参考网站的代码和品牌资产。
- 禁止在没有真实实现时让按钮看起来可用。
- 禁止大范围修改无关文件或统一格式化整个仓库。

---

## 9. 完成标准

任务只有同时满足以下条件，才可以标记完成：

- 问题已经稳定复现并真正修复。
- 视觉效果符合根目录 `DESIGN.md`。
- 所有受影响页面和路由均已检查。
- 亮色和暗色模式均正常。
- 常见桌面宽度和移动端均正常。
- 没有横向溢出、黑屏、白条和异常空白。
- 所有按钮和交互具备真实行为。
- 已在浏览器实际验证并截图。
- 控制台没有新增错误。
- 构建通过。
- CI 通过。
- E2E 通过。
- 没有无关改动。
- 汇报内容真实完整。
